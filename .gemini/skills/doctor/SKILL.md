# Doctor: Code Integrity & Quality Guardian

이 스킬은 코드 수정 시 발생할 수 있는 의도치 않은 부작용(Regression)을 방지하고, 시스템의 안정성을 극대화하기 위한 철저한 작업 가이드라인을 제공합니다.

## Core Mandates

모든 코드 수정 시 다음의 4대 원칙을 예외 없이 준수해야 합니다.

### 1. 전체 파일 쓰기 금지 (Surgical Changes Only)
- 파일의 크기와 상관없이 `write_file`을 통한 전체 덮어쓰기를 원칙적으로 금지합니다.
- 반드시 `replace` 도구를 사용하여 수정이 필요한 **함수나 블록 단위**로만 정밀하게 수정합니다.
- 도구 오류(예: `old_string` 매칭 실패) 발생 시 일괄 쓰기로 우회하지 말고, `read_file`을 통해 원인을 파악한 후 정밀 수정을 재시도합니다.

### 2. 수정 전 최신 상태 보존 및 검증
- 수정을 제안하기 직전에 반드시 `read_file`을 실행하여 대상 파일의 **현재 UI 구조와 로직을 다시 확인**합니다.
- 제안하는 코드가 파일의 최신 상태(들여쓰기, 최신 색상값, 라운딩 처리 등)를 그대로 반영하고 있는지 1:1로 대조합니다.

### 3. 영향도 평가 단계 의무화 (Impact Assessment)
- 로직(예: Throttling, 캐시 처리 등)을 추가/변경할 때, 이것이 리액트의 마운트/업데이트 사이클에서 다른 컴포넌트나 컨텍스트에 어떤 부작용을 줄지 다음 시나리오를 필수로 검토합니다:
    - **초기 로드(Hydration)** 시점의 동작
    - **상태 업데이트**에 따른 재렌더링 영향
    - **경쟁 상태(Race Condition)** 발생 가능성

### 4. 비파괴적 안정화 (Non-destructive Stabilization)
- 기능을 개선할 때 **기존에 잘 작동하던 UI나 로직은 절대 건드리지 않는 것**을 최우선으로 합니다.
- 개선 사항은 기존 코드와 조화롭게 공존해야 하며, 기존 기능을 파괴하면서 새로운 기능을 넣는 행위를 금지합니다.

## Workflow

1. **진단**: 수정이 필요한 부분의 현재 코드를 `read_file`로 정밀하게 읽습니다.
2. **평가**: 수정 시 발생할 수 있는 3가지 이상의 부작용 시나리오를 작성합니다.
3. **처방**: `replace` 도구를 사용하여 최소 단위의 코드를 수정합니다.
4. **검증**: 수정 후 기능이 정상 작동하는지 확인하고, 기존 UI/로직이 훼손되지 않았는지 재검토합니다.
